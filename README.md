# DAPRES Parent Portal — PWA

Built from `dapres_parent_portal_mockup_v3.html`, wired to the
`dr-alfredo-pio-de-roda` Firebase project (RTDB + Auth + FCM) — separate
from the `licenses-5c397` project ATSYS desktop uses for licensing, which
is untouched.

## ⚠️ One thing to check before deploying

You gave me `apiKey`, `authDomain`, `projectId`, `storageBucket`,
`messagingSenderId`, `appId`, `measurementId` — but **not** `databaseURL`.
I filled in the standard-format guess:

```
https://dr-alfredo-pio-de-roda-default-rtdb.firebaseio.com
```

That's correct *only* if the RTDB instance is in the default (US) region.
If you created it in another region (e.g. Singapore), the real URL looks
like `https://dr-alfredo-pio-de-roda-default-rtdb.asia-southeast1.firebasedatabase.app`
instead. Check Firebase console → Realtime Database → the URL shown at
the top, and fix it in all three places if it differs:
- `js/firebase-init.js` (`FIREBASE_CONFIG.databaseURL`)
- `firebase-messaging-sw.js` (same object, duplicated because service
  workers can't `import` the module)
- `atsys_v4_5_5/config.py` (`PARENT_FIREBASE_CONFIG["databaseURL"]`)

## What changed on the ATSYS (desktop) side

The actual bug behind "the dashboard doesn't make me codes" was that
`ParentLinkDashboardServer` (in `attendance_system_v4_5_4.py`) calls
`self.app.parent_sync.get_or_create_code(...)` and
`.list_codes_for_section(...)` — but `parent_sync_agent.py` never defined
either method. Both calls were wrapped in `try/except`, so it failed
completely silently — no error, just nothing happened when you clicked
Generate.

Fixed in the updated `parent_sync_agent.py`:
- Added `get_or_create_code()` / `list_codes_for_section()`, backed by a
  local ledger file (`~/.attendance_system/parent_link_codes.json`) —
  the dashboard reads that directly (instant, no network), and pushes
  each code to Firebase through the same durable offline-safe queue the
  scan sync already uses (so it still works if the code is generated
  while offline).
- Regenerating a code now also queues deleting the old one from Firebase,
  so a lost/replaced code actually stops validating.

Also rewired: `ParentSyncAgent` now points at `PARENT_FIREBASE_CONFIG`
(the new project) instead of the licensing `FIREBASE_CONFIG` it was
using before — added to `config.py`, with a fallback embedded directly
in `attendance_system_v4_5_4.py` in case `config.py` is ever missing.
**Licensing itself still uses `FIREBASE_CONFIG` (`licenses-5c397`),
untouched.**

The updated ATSYS files are in `../atsys_v4_5_5/` alongside this folder.
Only 3 files changed: `config.py`, `parent_sync_agent.py`,
`attendance_system_v4_5_4.py` — diff against your v4.5.4 bundle if you
want to see exactly what moved before rebuilding.

## RTDB data model

```
schools/DAPRES/
  attendance/{yyyy-mm-dd}/{student_id} = {status, time, grade, section, student_name}
  link_codes/{CODE}                    = {student_id, student_name, grade, section, expires_at}
parents/{uid}/
  students/{student_id} = {student_name, grade, section, linked_at}
  fcm_tokens/{token}    = true
```

`student_id` is the same composite slug ATSYS already builds
(`grade__section__name`), so nothing needs translating between the two
systems.

## Auth model (read this — it's the one real design call I made)

There's no school-wide parent login system, and building one (email
verification, password reset, etc.) is a lot of infrastructure for what
this needs. So the PWA uses **Firebase Anonymous Auth**: on first load it
silently signs in and gets a stable `uid` for that browser/device. The
**link code is the actual credential** — whoever has the code from the
school office can link that student on their device. Codes are **8-digit
numeric**, generated by the ATSYS dashboard.

First launch is gated: a parent can't browse any tab until they've
entered a valid code — there's no "skip for now." Once at least one
student is linked, the gate is gone for good on that device (adding more
students later uses the smaller "Link another student" modal in
Settings, same 8-digit code entry, not the full-screen gate).

This matches how the desktop dashboard already talks about codes ("read
or hand the code to a parent"). Trade-off: it's per-device, not
per-parent-identity — if a parent uses the portal on both their phone and
a tablet, they'd enter the code on each. If you'd rather have real parent
accounts (email/password or phone OTP) later, that's a bigger addition —
happy to build it if you want it, just say so.

## Alerts / push notifications — what's real vs. what needs a bit more

- **Alerts tab**: fully live. It diffs each realtime attendance snapshot
  against what this device last saw and turns new late/absent/on-time
  entries into alert rows, persisted locally.
- **Actual push notifications** (phone buzzes while the app is closed):
  the client-side half is done — `js/app.js` requests permission and
  registers an FCM token under `parents/{uid}/fcm_tokens`,
  `firebase-messaging-sw.js` handles the incoming push. But *sending* a
  push has to happen server-side (a closed browser tab can't push
  itself) — that's what `functions/index.js` does: it watches for new
  attendance writes and sends FCM to every parent who has that student
  linked. This needs the **Blaze (pay-as-you-go)** Firebase plan since
  Cloud Functions don't run on the free tier — deploy it if/when you're
  ready:
  ```
  cd functions && npm install && firebase deploy --only functions
  ```
  Until then, the Alerts tab still works fine (see above) — you just
  won't get a phone notification while the app isn't open.

## FCM VAPID key (needed before push works at all)

`js/firebase-init.js` has `FCM_VAPID_KEY = ""`. Get yours from Firebase
console → Project settings → Cloud Messaging → Web configuration → Web
Push certificates → generate a key pair, paste it in. Until it's set,
the app runs completely normally — it just skips push-token registration.

## Security rules

`database.rules.json` is a starting point:
- `schools/` stays fully open (read **and** write) — the desktop sync
  agent sends plain HTTPS with no auth token (same pattern as the rest
  of ATSYS's Firebase calls), so tightening this would break attendance
  sync and code generation from the desktop app.
- `parents/{uid}/` is locked to that uid's own anonymous auth session —
  real protection for linked-student lists, FCM tokens, and prefs.

Deploy with `firebase deploy --only database` (or push through the
Firebase console's Rules tab).

## Deploying the PWA itself

```
npm install -g firebase-tools   # if you don't have it
firebase login
firebase use dr-alfredo-pio-de-roda
firebase deploy --only hosting
```

Or just upload this folder's contents to whatever host serves
`dapres.dpdns.org` today (the mockup already referenced that domain for
the school logo) — it's a plain static site, no build step, nothing to
compile. `firebase-messaging-sw.js` **must** stay at the site root
(hard FCM requirement, not a preference).

## Icons

## Testing locally

Service workers and Firebase Auth both need HTTPS (or `localhost`) —
plain `file://` won't work. Easiest local test:
```
npx http-server . -p 8080
```
then open `http://localhost:8080`.
